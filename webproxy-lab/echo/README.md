# Echo Server Guide

이 문서는 현재 프로젝트의 `echo` 디렉터리에서 CS:APP 스타일의 echo 서버를
구현하고 테스트할 때 참고하는 작업 가이드다. 목적은 코드를 바로 제공하는 것이
아니라, 어떤 파일에 무엇을 구현하고 어떤 순서로 검증하면 되는지 빠르게
정리하는 데 있다.

`tiny`와 `proxy`를 구현하기 전에 echo 서버를 먼저 만드는 이유는 단순하다.
소켓 생성, 연결 수락, 바이트 단위 입출력, 종료 처리 같은 가장 기본적인 네트워크
흐름을 작은 프로그램으로 먼저 익힐 수 있기 때문이다. echo 서버가 안정적으로
동작하면, 이후 `tiny`와 `proxy`에서 필요한 연결 처리 흐름을 훨씬 쉽게 따라갈 수
있다.

## Directory Layout

현재 `echo` 디렉터리에는 아래 파일들이 준비되어 있다.

```text
echo/
├── Makefile
├── README
├── README.md
├── csapp.c
├── csapp.h
├── echo.c
└── echoserveri.c
```

파일 역할은 다음처럼 잡고 진행하면 된다.

- `echoserveri.c`: iterative echo 서버의 메인 프로그램
- `echo.c`: 연결된 클라이언트 소켓 디스크립터를 받아 echo 동작을 수행하는 함수 구현
- `csapp.c`, `csapp.h`: 교재에서 제공하는 robust I/O, 소켓 wrapper 등 공통 유틸
- `Makefile`: echo 서버 빌드 스크립트
- `README`: 필요하면 원문 메모나 교재 발췌 정리용으로 사용
- `README.md`: 구현 및 테스트 가이드 문서

## What To Implement

이번 단계에서는 보통 아래 구조를 목표로 잡으면 된다.

1. 서버가 포트 번호를 인자로 받는다.
2. listening socket을 생성하고 클라이언트 연결을 기다린다.
3. 연결이 들어오면 해당 연결을 처리한다.
4. 클라이언트가 보낸 바이트를 읽어서 그대로 다시 돌려준다.
5. 연결이 종료되면 다음 연결을 계속 받는다.

핵심 포인트는 "받은 데이터를 가공하지 않고 그대로 다시 보내는 것"이다.
이 동작이 안정적으로 되면 이후 HTTP 요청 라인을 읽는 `tiny`, 요청과 응답을
중계하는 `proxy`로 자연스럽게 이어진다.

## Recommended Build Shape

`Makefile`은 `tiny`와 루트 `Makefile`의 스타일을 참고해 아주 단순하게
구성하는 것이 좋다. 최소 목표는 다음과 같다.

- `make` 실행 시 `echoserveri` 바이너리 생성
- `csapp.c`와 `echoserveri.c`, 필요하면 `echo.c`를 함께 컴파일
- `make clean`으로 오브젝트 파일과 실행 파일 삭제

예를 들어 빌드 결과 파일명은 `echoserveri`로 두면 소스 파일명과 대응이
분명해서 이후 테스트 명령도 읽기 쉬워진다.

## Implementation Order

추천 순서는 아래와 같다.

1. `echo.c`에 echo 처리 함수의 입출력 흐름부터 정리한다.
2. `echoserveri.c`에서 `main`, 포트 인자 처리, listen socket 생성, accept 루프를 만든다.
3. `Makefile`에 실행 파일 타깃을 추가한다.
4. 빌드가 되면 로컬 수동 테스트부터 수행한다.
5. 예외 상황과 종료 동작을 점검한다.

처음부터 모든 예외를 다 처리하려고 하기보다, "한 클라이언트 연결이 정상적으로
echo 되는지"를 먼저 확인하는 편이 좋다.

## Port Selection

이 프로젝트에는 포트 선택을 도와주는 스크립트가 이미 있다.

- `../port-for-user.pl <userID>`: 사용자 기준으로 충돌 가능성이 낮은 포트 추천
- `../free-port.sh`: 현재 비어 있는 TCP 포트 탐색

예시:

```bash
cd ..
./free-port.sh
./port-for-user.pl your-id
```

실습 중에는 충돌을 피하기 위해 `tiny`, `proxy`, `echo`가 서로 다른 포트를
사용하는 것이 좋다.

## Manual Test Flow

echo 서버는 자동 채점기보다 먼저 수동 테스트로 감을 잡는 편이 훨씬 좋다.
가장 쉬운 흐름은 아래와 같다.

1. echo 서버를 한 터미널에서 실행한다.
2. 다른 터미널에서 `telnet`, `nc`, 또는 비슷한 TCP 클라이언트로 접속한다.
3. 텍스트를 입력한다.
4. 입력한 내용이 그대로 다시 돌아오는지 확인한다.
5. 여러 줄 입력, 빈 줄, 긴 줄, 연결 종료 상황을 확인한다.

예시 명령:

```bash
cd echo
make
./echoserveri 5000
```

다른 터미널:

```bash
telnet localhost 5000
```

또는:

```bash
nc localhost 5000
```

확인해야 할 기본 결과:

- `hello`를 입력하면 `hello`가 그대로 돌아와야 한다.
- 여러 줄을 보내면 각 줄이 빠지지 않고 다시 돌아와야 한다.
- 클라이언트가 종료된 뒤에도 서버 프로세스는 다음 연결을 계속 기다려야 한다.

## Suggested Test Cases

아래 항목을 순서대로 점검하면 충분하다.

### 1. Single Client Round Trip

- 클라이언트 1개로 접속
- 짧은 문자열 전송
- 동일한 문자열이 그대로 반환되는지 확인

### 2. Multiple Lines

- 여러 줄을 순서대로 입력
- 줄 단위로 깨지지 않고 그대로 echo 되는지 확인

### 3. Long Input

- 짧은 입력만 통과하는지 확인하지 말고 긴 문자열도 전송
- 버퍼 처리 실수가 없는지 확인

### 4. Client Disconnect

- 클라이언트를 강제로 종료하거나 EOF를 보냄
- 서버가 죽지 않고 다음 `accept`로 넘어가는지 확인

### 5. Reconnect

- 한 번 연결 종료 후 다시 접속
- 새 연결에서도 동일하게 동작하는지 확인

## Debugging Checklist

동작이 이상하면 아래부터 먼저 확인하면 된다.

- 포트 번호가 이미 다른 프로세스에서 사용 중인지
- `make` 결과로 실행 파일이 실제로 생성되었는지
- 서버가 `localhost`에서 대기 중인지
- 클라이언트 연결 직후 서버가 바로 종료되지 않는지
- 읽기 함수와 쓰기 함수가 같은 바이트 수를 기준으로 동작하는지
- EOF를 에러처럼 처리해 서버 전체가 종료되지 않는지

## Done Criteria

아래 조건을 만족하면 echo 서버 구현 단계는 거의 끝난 상태로 봐도 된다.

- `make`로 빌드된다.
- 서버가 포트를 받아 정상적으로 실행된다.
- 클라이언트 입력이 그대로 다시 반환된다.
- 클라이언트 종료 후에도 서버가 다음 연결을 받을 수 있다.
- 최소한 `telnet` 또는 `nc` 기준 수동 테스트를 통과한다.

## Next Step

echo 서버가 안정적으로 동작하면 다음으로 넘어갈 준비가 된 것이다.

- `tiny`: 요청 라인 파싱, 정적 파일 응답, CGI 실행
- `proxy`: 클라이언트 요청 읽기, 서버로 전달, 응답 중계, 이후 캐시와 동시성 확장

echo 단계에서 익힌 `listen -> accept -> read/write -> close` 흐름이 이후 구현의
기본 뼈대가 된다.
