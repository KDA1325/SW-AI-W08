# csapp.c는 무엇을 하는 파일인가

이 코드가 해결하려는 문제가 무엇인지 먼저 보겠습니다.

`csapp.c`는 CS:APP 책에서 제공하는 **시스템 프로그래밍용 유틸리티 라이브러리 구현 파일**입니다.  
쉽게 말하면, 리눅스 시스템 콜과 네트워크 함수를 직접 다룰 때 반복해서 써야 하는 **에러 처리**, **래퍼 함수**, **안전한 입출력**, **소켓 생성/연결 보조 함수**를 한곳에 모아둔 파일입니다.

실무에서는 이렇게 생각하시면 됩니다.

- 시스템 콜은 강력하지만 거칠다
- 성공하면 좋지만, 실패했을 때 매번 `if`, `errno`, `strerror()`를 직접 처리해야 한다
- 그래서 `csapp.c`는 "거친 시스템 콜을 조금 더 안전하고 일관되게 감싼 도구 모음" 역할을 한다

즉, 이 파일은 애플리케이션의 비즈니스 로직을 담는 파일이 아니라,  
그 위에서 프로그램을 더 편하게 만들 수 있게 해주는 **기반 도구 상자**입니다.

---

# 먼저 큰 그림을 봅시다

`csapp.c`는 크게 보면 아래 흐름으로 구성됩니다.

1. 에러를 일관되게 출력하는 함수
2. 시스템 콜을 감싸는 래퍼 함수
3. 시그널 안전 입출력 함수
4. 일반적인 파일/메모리/디렉터리/소켓 래퍼 함수
5. Robust I/O(`rio`) 패키지
6. 클라이언트/서버 연결을 쉽게 여는 헬퍼 함수

이걸 하나의 공장으로 비유해보겠습니다.

- **에러 함수**는 사고가 났을 때 경보를 울리는 장치입니다.
- **래퍼 함수**는 위험한 기계를 덮개 씌워서 쓰기 쉽게 만든 장치입니다.
- **RIO 패키지**는 데이터가 중간에 끊기거나 덜 읽히는 문제를 줄여주는 안전 장비입니다.
- **open_clientfd / open_listenfd**는 네트워크 연결 공정을 한 번에 처리해주는 자동화 라인입니다.

이제 왜 이런 구조로 가는지 이해가 되시죠?

# csapp.c가 필요한 이유

자, 이 기술이 없던 시절로 돌아가봅시다.

우리가 `read()`, `write()`, `open()`, `socket()`, `connect()`를 직접 쓴다고 해보겠습니다.

문제는 뭘까요?

1. 에러 처리 코드가 너무 많이 반복됩니다.
2. 어떤 함수는 실패 시 `-1`, 어떤 함수는 `NULL`, 어떤 함수는 다른 값을 반환합니다.
3. `read()`나 `write()`는 한 번 호출했다고 원하는 바이트 수를 전부 처리해주지 않을 수 있습니다.
4. 소켓 연결은 `getaddrinfo`, `socket`, `connect`, `bind`, `listen`을 순서대로 다 써야 해서 실수하기 쉽습니다.

그래서 `csapp.c`는 이런 반복과 위험을 줄이기 위해 등장합니다.

핵심은 이것입니다.

- **에러 처리 일관화**
- **반복 코드 제거**
- **네트워크/입출력 실수를 줄이는 안전 장치 제공**

# 1. 에러 처리 함수는 무슨 역할인가

가장 먼저 나오는 `unix_error`, `posix_error`, `gai_error`, `app_error`는  
**오류를 어떤 방식으로 출력할지 통일하는 함수**입니다.

예를 들어:

- `unix_error`: `errno` 기반 시스템 콜 오류 출력
- `posix_error`: pthread 같은 POSIX 함수의 반환 코드 기반 오류 출력
- `gai_error`: `getaddrinfo()` 계열 오류 출력
- `app_error`: 일반 애플리케이션 오류 출력

왜 필요할까요?

시스템 프로그래밍에서는 실패 원인을 빨리 알아야 합니다.  
매번 아래처럼 쓰면 너무 반복됩니다.

```c
if (open(...) < 0) {
    fprintf(stderr, "...: %s\n", strerror(errno));
    exit(0);
}
```

그래서 이 부분을 함수로 빼서 재사용하는 겁니다.

실무에서는 이렇게 생각하시면 됩니다.

- 에러 메시지 형식이 통일되면 디버깅이 쉬워집니다.
- 실패 시 바로 종료할지, 상위로 넘길지 정책을 한 곳에서 정리하기 좋습니다.

---

# 2. 프로세스 제어 래퍼는 무엇인가

`Fork`, `Execve`, `Wait`, `Waitpid`, `Kill` 같은 함수들은  
원래 있는 시스템 콜을 그대로 감싼 함수입니다.

예를 들어 `Fork()`는 내부적으로 `fork()`를 호출합니다.  
차이는 뭘까요?

- 원래 `fork()`는 실패하면 `-1`을 반환합니다.
- `Fork()`는 실패하면 `unix_error("Fork error")`를 호출해서 즉시 에러를 처리합니다.

즉,

- **원본 함수**: 실패 여부를 호출자가 직접 처리해야 함
- **래퍼 함수**: 실패 처리를 라이브러리가 대신 해줌

이게 진짜 중요합니다.

이 파일 전체의 설계 철학이 바로 이것입니다.

**"시스템 콜은 그대로 쓰되, 실수하기 쉬운 부분은 감싸서 단순하게 만든다."**

---

# 3. 시그널 래퍼는 왜 따로 있을까

시그널은 비동기적으로 들어옵니다.  
즉, 프로그램이 뭘 하고 있든 중간에 끼어들 수 있습니다.

그래서 시그널 관련 코드는 더 조심해야 합니다.

`Signal()` 함수 안을 보면 `sigaction`을 사용하고,  
`SA_RESTART` 플래그를 설정합니다.

이게 왜 중요할까요?

시그널이 들어오면 어떤 시스템 콜은 중간에 끊길 수 있습니다.  
그때 자동으로 재시작되게 도와주는 게 `SA_RESTART`입니다.

즉, `Signal()`은 단순히 시그널 핸들러 등록이 아니라  
**실무적으로 더 안전한 방식으로 시그널 처리 기본값을 맞춰주는 래퍼**입니다.

---

# 4. SIO는 왜 따로 있나

자, 여기서 질문이 나옵니다.

"그냥 `printf()` 쓰면 되지, 왜 `sio_puts`, `sio_putl` 같은 게 필요하지?"

좋은 질문입니다.

이건 **signal-safe I/O** 때문입니다.

시그널 핸들러 안에서는 아무 함수나 호출하면 안 됩니다.  
`printf()` 같은 함수는 내부 상태를 사용하기 때문에 안전하지 않을 수 있습니다.

그래서 `csapp.c`는 시그널 핸들러 안에서도 상대적으로 안전하게 쓸 수 있는  
간단한 출력 함수 묶음을 따로 제공합니다.

구성은 이렇게 됩니다.

- `sio_reverse`: 문자열 뒤집기
- `sio_ltoa`: long 값을 문자열로 바꾸기
- `sio_strlen`: 길이 계산
- `sio_puts`: 문자열 출력
- `sio_putl`: 숫자 출력
- `sio_error`: 에러 출력 후 종료

즉, 이 파트는 **시그널 환경에서 최소한의 출력 기능만 제공하는 작은 입출력 패키지**입니다.

# 5. Unix I/O 래퍼는 어떻게 읽으면 될까

이 부분은 패턴이 거의 같습니다.

예를 들어 `Open`, `Read`, `Write`, `Close`를 보면:

1. 원래 시스템 콜 호출
2. 실패 여부 확인
3. 실패 시 `unix_error`
4. 성공 시 결과 반환

예를 들어 `Read()`는 이렇게 이해하시면 됩니다.

- `read(fd, buf, count)` 호출
- 반환값이 음수면 에러
- 아니면 그대로 반환

즉, 이 함수들의 핵심 가치는 "새로운 기능"이 아니라  
**호출 패턴을 통일한다는 것**입니다.

실제로 현업에서도 이런 래퍼를 많이 둡니다.

왜?

- 매번 에러 처리 코드 쓰기 싫어서
- 로그 형식을 통일하려고
- 정책을 한 곳에서 바꾸려고

# 6. 메모리, 디렉터리, 표준 입출력 래퍼는 왜 같은 파일에 있나

처음 보면 좀 이상할 수 있습니다.

- 파일 함수도 있고
- 디렉터리 함수도 있고
- 메모리 할당도 있고
- 소켓도 있고
- 스레드도 있습니다

왜 한 파일에 몰아놨을까요?

이 파일의 목적은 "한 가지 기능"이 아니라  
**시스템 프로그래밍 실습에 자주 쓰는 안전한 공용 유틸을 한곳에 모으는 것**이기 때문입니다.

즉, `csapp.c`는 프레임워크라기보다  
**교육용 + 실습용 표준 유틸 라이브러리**에 가깝습니다.

# 7. 소켓 래퍼는 네트워크 프로그래밍을 어떻게 단순화하나

`Socket`, `Bind`, `Listen`, `Accept`, `Connect`는  
네트워크 프로그래밍의 핵심 시스템 콜입니다.

원래는 이런 함수들을 직접 호출하면서 전부 에러 처리를 해야 합니다.

그런데 `csapp.c`는 이걸 래퍼로 감싸서 아래처럼 읽기 쉽게 만듭니다.

- `Socket(...)`: 소켓 생성
- `Bind(...)`: 주소와 포트 묶기
- `Listen(...)`: 연결 대기 상태로 만들기
- `Accept(...)`: 클라이언트 연결 받기
- `Connect(...)`: 서버에 연결하기

이 함수들은 기능 자체보다도 **네트워크 코드의 가독성**을 높여줍니다.

---

# 8. 주소 변환 함수는 왜 중요한가

`Getaddrinfo`, `Getnameinfo`, `Inet_ntop`, `Inet_pton`은  
주소를 문자열과 네트워크 표현 사이에서 바꾸는 함수입니다.

예전에는 `gethostbyname`, `inet_aton`, `inet_ntoa` 같은 함수들을 많이 썼는데,  
이 파일에서도 주석으로 나오듯이 그 방식은 오래됐고 스레드 안전하지 않은 부분이 있습니다.

그래서 현대적인 방식은:

- `getaddrinfo`
- `getnameinfo`

입니다.

이게 왜 좋을까요?

- IPv4, IPv6를 더 유연하게 처리할 수 있습니다.
- 프로토콜 독립적인 코드 작성이 쉬워집니다.
- 재진입성과 스레드 안전성이 더 좋습니다.

# 9. RIO 패키지가 핵심입니다

이 파일에서 가장 중요한 부분 중 하나가 `rio`입니다.

자, 왜 이게 필요할까요?

우리가 `read(fd, buf, n)`을 호출했다고 해서  
항상 `n`바이트를 한 번에 읽는 게 아닙니다.

마찬가지로 `write(fd, buf, n)`도  
항상 `n`바이트를 한 번에 다 쓰는 게 아닙니다.

여기서 초보자들이 많이 실수합니다.

그래서 `csapp.c`는 Robust I/O 패키지를 제공합니다.

## RIO의 큰 그림

### 1. unbuffered robust I/O
- `rio_readn`
- `rio_writen`

이 함수들은 "원하는 바이트 수만큼 끝까지 읽거나 쓸 때까지" 반복합니다.

### 2. buffered robust I/O
- `rio_read`
- `rio_readinitb`
- `rio_readnb`
- `rio_readlineb`

이 함수들은 내부 버퍼를 두고,  
특히 텍스트 줄 단위 읽기를 편하게 해줍니다.

---

# 10. rio_readn / rio_writen은 어떻게 동작하나

먼저 가장 단순한 구조부터 보겠습니다.

`rio_readn(fd, buf, n)`의 목표는 딱 하나입니다.

**"가능하면 n바이트를 끝까지 읽자."**

흐름은 이렇습니다.

1. 아직 읽어야 할 바이트 수 `nleft`를 둡니다.
2. `read()`를 호출합니다.
3. 일부만 읽히면 남은 양을 줄입니다.
4. 포인터를 앞으로 이동합니다.
5. 남은 양이 0이 될 때까지 반복합니다.
6. EOF를 만나면 중간까지 읽은 만큼 반환합니다.
7. 시그널 때문에 끊긴 경우(`EINTR`)는 다시 시도합니다.

`rio_writen()`도 똑같습니다.

**"가능하면 n바이트를 끝까지 쓰자."**

이게 왜 중요하냐면,  
네트워크 프로그래밍에서는 "한 번 호출 = 전부 처리"라고 생각하면 안 되기 때문입니다.

이것만 기억하세요.

- `read/write`는 부분 성공할 수 있다
- `rio_readn/rio_writen`은 그 문제를 줄여준다

---

# 11. rio_readlineb는 왜 echo 서버에서 중요할까

`rio_readlineb()`는 줄 단위 텍스트를 읽습니다.

예를 들어 echo 서버에서 클라이언트가 한 줄 보내면,  
서버는 그 한 줄을 읽고 그대로 다시 보내면 됩니다.

이때 `rio_readlineb()`를 쓰면 편합니다.

흐름은 이렇습니다.

1. 내부 버퍼에서 1바이트씩 가져옵니다.
2. `\n`을 만나면 한 줄이 끝났다고 봅니다.
3. 문자열 끝에 `0`을 붙여 C 문자열로 만듭니다.
4. 읽은 길이를 반환합니다.

즉, 이 함수는 텍스트 프로토콜에서 굉장히 유용합니다.

- echo 서버
- tiny 웹 서버
- HTTP 헤더 읽기

이런 데서 자주 씁니다.

---

# 12. rio_read는 내부 버퍼 관리자입니다

`rio_read()`는 바깥에서 직접 많이 부르는 함수는 아닙니다.  
내부 헬퍼에 가깝습니다.

이 함수가 하는 일은:

- 내부 버퍼에 데이터가 없으면 `read()`로 다시 채우고
- 거기서 필요한 만큼만 사용자 버퍼로 복사하는 것

즉, `rio_readnb()`와 `rio_readlineb()`가  
더 편하게 동작할 수 있도록 뒤에서 버퍼를 관리해주는 엔진입니다.

비유하면 이렇습니다.

- `rio_read()`는 창고 관리자
- `rio_readlineb()`는 손님이 한 줄 주문하는 창구
- `rio_readnb()`는 손님이 정해진 양만큼 주문하는 창구

---

# 13. open_clientfd와 open_listenfd는 네트워크 진입점입니다

이제 네트워크 쪽 큰 그림을 봅시다.

## open_clientfd
이 함수의 목표는:

**주어진 호스트와 포트에 연결 가능한 클라이언트 소켓 하나를 열어주는 것**

흐름은 이렇습니다.

1. `getaddrinfo()`로 가능한 서버 주소 후보를 구합니다.
2. 후보를 하나씩 순회합니다.
3. 각 후보마다 `socket()`으로 소켓을 만듭니다.
4. `connect()`를 시도합니다.
5. 성공하면 그 소켓을 반환합니다.
6. 실패하면 닫고 다음 후보를 시도합니다.

즉, "연결 가능한 주소를 찾아가며 하나씩 시도하는 함수"입니다.

## open_listenfd
이 함수의 목표는:

**주어진 포트에서 클라이언트 연결을 받을 수 있는 리슨 소켓을 열어주는 것**

흐름은 이렇습니다.

1. `getaddrinfo()`로 바인드 가능한 주소 후보를 구합니다.
2. 후보를 하나씩 순회합니다.
3. `socket()`으로 소켓을 만듭니다.
4. `setsockopt(... SO_REUSEADDR ...)`로 주소 재사용 옵션을 켭니다.
5. `bind()`를 시도합니다.
6. 성공하면 `listen()`으로 리슨 상태로 바꿉니다.
7. 최종적으로 리슨 소켓 fd를 반환합니다.

이 두 함수는 Tiny 서버와 Echo 서버를 이해할 때 핵심입니다.

왜?

- 클라이언트는 `open_clientfd()`로 연결을 시작하고
- 서버는 `open_listenfd()`로 연결 받을 준비를 하기 때문입니다.

# 17. 마지막 정리

이제 전체 흐름을 짧게 다시 정리해보겠습니다.

- 에러 처리 함수가 실패 정책을 통일합니다.
- 래퍼 함수가 시스템 콜 호출을 단순하게 만듭니다.
- RIO 패키지가 읽기/쓰기의 불완전함을 보완합니다.
- `open_clientfd`, `open_listenfd`가 네트워크 연결 진입점을 단순화합니다.
- `echo`, `tiny` 같은 프로그램은 이 파일 위에서 더 짧고 안전하게 작성됩니다.

결국 이 파일의 핵심은 이것입니다.

**복잡한 시스템 콜 세계를 조금 더 안전하고 예측 가능하게 만들어주는 기반 유틸리티**

느낌이 오시죠?
